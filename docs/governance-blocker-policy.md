# Folio 开发受阻规则（MVP）

## 1. 目标

让“卡住了”变成一个可管理状态，而不是一句无效同步。

## 2. blocker 定义

以下情况可认定为 blocker：

- 缺少关键决策，无法继续实现
- 依赖外部接口/仓库/权限尚未就绪
- 环境、凭证、工具链导致关键路径无法验证
- 上游需求持续变化，当前实现无稳定目标
- 评审或协作方超过约定时间未反馈，导致无法推进

## 3. 进入 blocked 时必须补充的信息

进入 `blocked` 时，issue 或评论中必须写明：

- **阻断原因**：到底卡在哪
- **影响范围**：阻断的是开发、测试、发布还是评审
- **需要谁处理**：责任方是谁
- **解除条件**：满足什么就能恢复
- **下次同步时间**：何时再次检查

## 4. 处理时限建议

- 普通 blocker：24 小时内至少同步一次
- 高优 blocker：当天内升级反馈
- 连续 2 次同步仍未解除：升级为管理问题或拆出依赖 issue

## 5. 行为规则

- blocker 不等于停摆。若主路径被阻断，应优先寻找可并行子项
- 若 blocker 超过 2 个工作日，建议拆分“阻断依赖 issue”
- 无法描述清楚阻断原因的，不应直接标记为 `blocked`
- blocker 解除后，应明确回到 `ready` 或 `in_progress`

## 6. blocker 记录模板

```text
阻断原因：
影响范围：
责任方：
解除条件：
下次同步时间：
备用推进方案（可空）：
```
