# Folio 开发流程规范（MVP）

> 目标：让 Folio 在早期阶段就具备“能协作、能追踪、能自动化衔接”的基本治理能力。

## 1. 适用范围

本规范适用于 Folio 仓库内的功能开发、缺陷修复、文档治理、工程改造与自动化建设。

## 2. 工作原则

- **Issue 先行**：除极小修正外，开发工作原则上先有 issue，再开分支、再开发。
- **小步提交**：单个 PR 应聚焦单一目标，避免“顺手大改”。
- **中文优先**：讨论、文档、模板优先中文，必要时补英文。
- **可验证**：提交与 PR 必须写清验证方式。
- **先 MVP，后自动化**：先把规则和状态固化，再逐步接自动编排。

## 3. 标准流程

### 3.1 需求进入

1. 提出需求 / 缺陷 / 任务
2. 创建 issue
3. 完成初步分类（feature / bug / chore / governance 等）
4. 判断优先级与负责人
5. 进入 `triaged` 或 `ready`

### 3.2 开发启动

1. 从 issue 创建分支
2. 明确范围、验收标准、风险
3. 如涉及设计变更，先在 issue 中确认
4. 开始开发

### 3.3 开发进行中

开发过程中应持续维护：

- issue 状态
- blocker 情况
- 范围是否发生变更
- 验证记录是否充分

### 3.4 提交与 PR

提交 PR 前至少满足：

- 变更范围明确
- 关联 issue
- 写明为什么改
- 写明如何验证
- 列出风险与兼容性影响

### 3.5 合并与收尾

PR 合并后应：

- 回写 issue 结果
- 更新相关文档
- 若有遗留项，拆成新 issue

## 4. 分支建议

建议分支命名：

- `feat/<issue-number>-<slug>`
- `fix/<issue-number>-<slug>`
- `chore/<issue-number>-<slug>`
- `governance/<issue-number>-<slug>`

示例：

- `governance/2-mvp-system`
- `feat/18-route-loader`

## 5. PR 规模建议

- **推荐**：单 PR 聚焦 1 个 issue
- **允许**：1 个根 issue 下带 2~4 个紧密相关子项
- **避免**：无关重构与需求混在同一个 PR

## 6. 例外处理

以下情况可先改后补 issue，但必须在 PR 中补齐说明：

- typo / 文案错字
- 不影响行为的注释修正
- 阻断线上或核心开发链路的紧急热修

## 7. 与后续自动化的关系

本规范为后续以下能力提供基础：

- 基于标签和状态的自动提醒
- 基于 issue / PR 字段完整性的自动校验
- 基于状态机的自动分派、自动催办
- 后续自动 merge / release 的门禁基础
