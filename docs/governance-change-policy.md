# Folio 需求变更规则（MVP）

## 1. 目标

避免开发过程中需求漂移、口头加活、边做边改导致的范围失控。

## 2. 变更分级

### A. 微小变更
不改变目标与验收标准，例如：
- 文案调整
- 命名优化
- 小型结构整理

处理方式：
- 允许直接并入当前 issue / PR
- 在 PR 描述中补充说明即可

### B. 范围内变更
不改变大目标，但会影响实现方案、测试点或交付清单，例如：
- 增加一个必要校验
- 增补文档/测试要求

处理方式：
- 必须在 issue 或 PR 评论中留痕
- 更新验收标准
- 若影响评审，通知 reviewer

### C. 范围外变更
会显著扩大工作量、延长周期或改变阶段目标，例如：
- 从“加检查”升级为“自动执行”
- 从“文档规范”升级为“完整自动编排平台”

处理方式：
- 不得直接塞进当前 PR
- 必须新开 issue 或拆分子任务
- 由负责人确认是否进入当前里程碑

## 3. 变更判定标准

满足任一项，优先按“范围外变更”处理：

- 预计新增工作量 > 30%
- 引入新的系统边界或新模块
- 需要新增流程、权限或自动化行为
- 改变原验收标准

## 4. 执行规则

- 所有变更必须留痕，禁止只在口头或私聊里生效
- 改变验收标准时，必须同步更新 issue
- 已进入 `in_review` 的 PR，原则上不再接收范围外变更
- 若变更来自上游决策，需记录“来源、原因、影响、处理结论”

## 5. 推荐记录模板

```text
变更类型：微小 / 范围内 / 范围外
变更来源：
变更原因：
影响范围：
是否改验收标准：是 / 否
处理结论：并入当前 / 拆新 issue / 延后
```
