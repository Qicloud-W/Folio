# Security / Risk Check Round 2

## 测试2结论

已完成首轮轻量安全检查，当前风险级别：**低到中**，适合作为 MVP 骨架继续推进，但不能把现在这版误判成可直接上生产。

## 检查项与结果

### 1. 错误信息泄漏

- 当前 `public/index.php` / `Kernel` 没有显式异常捕获。
- 如果后续出现 `JSON_THROW_ON_ERROR`、文件加载或业务异常，可能直接把 PHP 默认错误抛给外部。
- 结论：**存在风险，需下一轮补统一异常响应层**。

### 2. 配置边界

- `.env` 仅做简单 `KEY=VALUE` 解析。
- 不支持复杂转义、多行值、注释尾随等高级语法。
- 优点是简单；缺点是边界较脆。
- 结论：**MVP 可接受，但需明确文档边界，避免错误配置引发异常行为**。

### 3. 输入/方法风险

- 当前只注册 `GET` 路由。
- 未命中方法会统一落入 404，而不是 405 Method Not Allowed。
- 这不算严重漏洞，但会影响接口语义与调试体验。
- 结论：**建议下一轮区分路径不存在与方法不允许**。

### 4. 编码风险

- `Response` 默认输出 `application/json; charset=utf-8`，这是对的。
- JSON 使用 `JSON_UNESCAPED_UNICODE`，中文输出可读。
- 语言文件当前为 PHP 数组，风险可控。
- 结论：**当前编码策略正常**。

### 5. 路由与文件加载

- 目前路由在 `Kernel` 内直接声明，避免了动态 include 带来的额外输入面。
- 但尚未接入 `routes/api.php`，后续引入时需限制返回结构，别搞成任意执行入口。

## 优先修复建议

1. 增加全局异常捕获，统一返回 JSON 错误结构。
2. 增加 `APP_DEBUG=false` 时的错误脱敏策略。
3. route dispatcher 区分 404 / 405。
4. 给 `Env` 和 `ConfigLoader` 补最小单元测试。
